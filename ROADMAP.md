# ROADMAP.md

План развития ProbablyFresh с фокусом на production-ready контур.

## Принципы

- Не ломать текущий pipeline и backward compatibility.
- Развивать систему итеративно: сначала надежность и качество данных, затем функциональность.
- Любое изменение должно иметь наблюдаемость и проверку.

## Приоритет 1: ingestion и schema contracts

### 1.1 Управляемый ingestion из UI

Цель:
- добавить кнопку загрузки данных в UI;
- принимать файл через backend API;
- запускать асинхронную обработку batch.

Минимум:
- endpoint загрузки (`POST /api/imports`),
- статус batch (`GET /api/imports/{id}`),
- отчёт ошибок (`GET /api/imports/{id}/errors`).

### 1.2 Контракты схем (Schema Contracts)

Цель:
- исключить «тихие» поломки при изменении формата входных данных.

Шаги:
- ввести `schema_version`;
- хранить правила маппинга (preset);
- валидировать обязательные поля, типы, enum, диапазоны;
- новый/неизвестный формат отправлять в карантин, а не в MART.

## Приоритет 2: Data Quality 2.0

### 2.1 Расширенные проверки

- валидация формата ID и бизнес-диапазонов;
- cross-table проверки (согласованность customer/store/product);
- контроль аномалий объема и свежести данных.

### 2.2 Таблица ошибок качества

Добавить `dq_errors`:
- `entity`, `error_code`, `error_message`, `raw_payload`, `run_id`, `event_time`.

Польза:
- прозрачный разбор «почему строка отбракована»;
- база для алертов и SLA.

## Приоритет 3: observability и алертинг

### 3.1 Метрики pipeline

- длительность задач,
- throughput по этапам,
- success/fail rate,
- lag ingestion.

### 3.2 Унификация метрик UI/Grafana

Сейчас возможны различия между RAW/MART/snapshot метриками.

Задача:
- зафиксировать единые определения KPI;
- явно маркировать источник метрики в UI и Grafana.

## Приоритет 4: безопасность и управление доступом

### 4.1 RBAC

- роли для запуска pipeline действий;
- разграничение доступа к destructive операциям.

### 4.2 Audit Trail

- расширить аудит действий пользователя,
- хранить: кто/когда/что запустил, с какими параметрами и результатом.

### 4.3 Секреты

- переход от хранения чувствительных данных в plain `.env` к менеджеру секретов.

## Приоритет 5: CI/CD и тестирование

### 5.1 Автотесты

- smoke-test контура в CI,
- contract-tests для входных схем,
- SQL sanity checks для init-скриптов.

### 5.2 Delivery

- автоматическая проверка миграций Django,
- линт и type-check для backend/frontend,
- контроль артефактов сборки.

## Приоритет 6: развитие витрины и UX

- расширение ETL-признаков и версионирование фич;
- отдельный MART dashboard;
- планировщик quality snapshots;
- улучшение операторского интерфейса (массовые операции, сохраненные сценарии, export-run traceability).

## Безопасные следующие шаги (рекомендуемый порядок)

1. Внедрить `schema_version` + базовый валидатор импорта.
2. Добавить `dq_errors` и отчеты по причинам отбраковки.
3. Унифицировать definitions метрик между UI и Grafana.
4. Добавить RBAC для action-endpoints и destructive jobs.
5. Подключить CI smoke-test с Docker-only сценарием.

## Критерий готовности к production

Система считается production-ready, когда:
- ingestion защищен контрактами и не ломает контур при schema mismatch;
- quality-контроль покрывает дубликаты, invalid и свежесть;
- метрики и алерты единообразны и интерпретируемы;
- есть управляемый доступ (RBAC), аудит и CI/CD-проверки.

1) Что реализовано (итог по проекту)
====================================================================

Реализован полный MVP-пайплайн данных:

1. Генерация демо-данных в JSON на диск:
   - stores: 45 файлов (30 Almost + 15 Maybe)
   - products: 100 файлов (20 на категорию, 5 категорий)
   - customers: >=45 (фактически 175 при текущей конфигурации генератора)
   - purchases: >=200 (фактически 200)

2. Загрузка JSON в MongoDB:
   - коллекции: stores, products, customers, purchases
   - upsert по ключам:
     stores.store_id
     products.id
     customers.customer_id
     purchases.purchase_id
   - логируется processed/inserted/updated

3. Ingestion MongoDB -> Kafka -> ClickHouse RAW:
   - Kafka топики:
     probablyfresh.stores
     probablyfresh.products
     probablyfresh.customers
     probablyfresh.purchases
   - В ClickHouse создан RAW слой:
     probablyfresh_raw.stores_raw
     probablyfresh_raw.products_raw
     probablyfresh_raw.customers_raw
     probablyfresh_raw.purchases_raw
   - Также созданы Kafka Engine таблицы и Materialized Views Kafka -> RAW

4. Нормализация и шифрование PII до ClickHouse:
   - для customers и purchases перед отправкой в Kafka:
     email: trim + lower
     phone: приведение к E.164-подобному виду
   - далее шифрование через Fernet (cryptography), ключ из FERNET_KEY
   - в RAW хранится уже зашифрованный email/phone

5. MART слой в ClickHouse:
   - база probablyfresh_mart
   - таблицы на ReplacingMergeTree(ingested_at):
     stores_mart
     products_mart
     customers_mart
     purchases_mart
   - Materialized Views RAW -> MART с очисткой/валидацией:
     * trim/lower для строковых ключей
     * фильтрация пустых ключей
     * валидация дат (birth_date <= today, purchase_dt <= now)
   - таблица качества:
     mart_quality_stats
     с метриками total/inserted/invalid/duplicates/duplicates_ratio

6. Grafana:
   - автоматическое provisioning datasource ClickHouse
   - автоматическое provisioning dashboard
   - dashboard "ProbablyFresh RAW Overview":
     Stores count
     Purchases count
     Stores by network
     метрики качества MART (duplicates/invalid/ratio)
   - настроено provisioning alerting:
     * contact point Telegram: rwss_grafana_bot
     * notification policy
     * alert rule Duplicates_ratio (>0.5, eval 1m, for 1m)
     * telegram template

7. PySpark ETL (jobs/features_etl.py):
   - читает customers_mart и purchases_mart через JDBC (ClickHouse)
   - считает customer-фичи (0/1):
     recurrent_buyer
     delivery_user
     bulk_buyer
     low_cost_buyer
     prefers_cash
     prefers_card
     weekend_shopper
     weekday_shopper
     night_shopper
     morning_shopper
   - сохраняет один CSV (coalesce(1))
   - грузит в S3/MinIO как analytic_result_YYYY_MM_DD.csv

8. Оптимизирован Docker-run ETL:
   - создан кастомный app image: docker/app/Dockerfile
   - зависимости (включая pyspark) ставятся на этапе build, а не каждый запуск
   - в образ заранее скачивается clickhouse-jdbc-0.9.6-all-dependencies.jar
   - spark-submit использует локальный fat-jar (без постоянной подкачки с Maven)

====================================================================
2) Архитектура и поток данных
====================================================================

[JSON files] -> [MongoDB] -> [Kafka topics] -> [ClickHouse RAW] -> [ClickHouse MART] -> [Grafana + Alerts]
                                                              \
                                                               -> [PySpark ETL] -> [S3/MinIO CSV]

Пояснение:
- RAW накапливает события как пришли.
- MART очищает/валидирует/дедуплицирует (на чтении через FINAL или после merge).
- Duplicates рассчитываются в mart_quality_stats как snapshot по RAW.

4) Подробно по ключевым файлам
====================================================================

4.1 Генерация данных
Файл: src/generator/generate_data.py

Что делает:
- Берет seed из --seed или SEED env (default 42).
- Очищает только *.json в data/stores, data/products, data/customers, data/purchases.
- Генерирует:
  stores (45 шт)
  products (100 шт)
  customers (2..5 на магазин => обычно 175)
  purchases (200)
- В purchases delivery_address пишется только если is_delivery=true.
- Делает self-check:
  stores == 45
  products == 100
  ровно 20 на категорию
  customers >= 45
  purchases >= 200
- В конце печатает итоговые количества.

4.2 Загрузка в Mongo
Файл: src/loader/load_to_mongo.py

Что делает:
- Читает MONGO_URI и MONGO_DB из env.
- Сканирует data/stores, data/products, data/customers, data/purchases.
- Пишет в коллекции stores/products/customers/purchases.
- Upsert по ключам (см. выше).
- Логирует processed/inserted/updated.

4.3 Producer Mongo -> Kafka
Файл: src/streaming/produce_from_mongo.py

Что делает:
- Режим --once.
- Читает документы Mongo по коллекциям.
- Для customers/purchases:
  * рекурсивно ищет email/phone
  * normalizes
  * encrypts через Fernet
- Публикует JSON в соответствующие топики.
- Логирует количество отправленных записей.

4.4 Нормализация и крипто
Файлы:
- src/probablyfresh/core/normalization.py
- src/probablyfresh/core/crypto_utils.py

Что делают:
- normalize_email: trim + lower
- normalize_phone: очистка нецифровых символов, базовые правила РФ
- Encryptor: Fernet шифрование строки
- generate_key(): генерация ключа Fernet

4.5 SQL инициализация RAW
Файл: docker/clickhouse/init/01_init.sql

Содержит:
- CREATE DATABASE probablyfresh_raw
- CREATE TABLE *_raw (MergeTree)
- CREATE TABLE *_kafka (Kafka Engine)
- CREATE MATERIALIZED VIEW mv_*_raw (Kafka -> RAW)

4.6 SQL инициализация MART
Файл: docker/clickhouse/init/02_mart.sql

Содержит:
- CREATE DATABASE probablyfresh_mart
- CREATE TABLE *_mart (ReplacingMergeTree(ingested_at))
- CREATE MATERIALIZED VIEW mv_*_to_mart (RAW -> MART)
- CREATE TABLE mart_quality_stats
- INSERT snapshot метрик качества по каждой сущности

4.7 PySpark ETL
Файл: jobs/features_etl.py

Что делает:
- Загружает env через python-dotenv.
- Строит SparkSession:
  приоритет local JDBC jar (CLICKHOUSE_JDBC_JAR),
  fallback на spark.jars.packages.
- JDBC чтение таблиц purchases_mart/customers_mart.
- Агрегация фич и бинаризация (int 0/1).
- left join с customers, чтобы клиенты без покупок имели 0-фичи.
- Выгрузка CSV в tmp, поиск part-*.csv, upload в S3/MinIO через boto3.

4.8 Docker app image
Файл: docker/app/Dockerfile

Что делает:
- База python:3.12-slim

- Это убирает повторную установку pyspark/jdbc на каждом запуске.

7) Команды полного запуска с нуля (Docker-only)
====================================================================

Официальный runbook лежит в файле:
- ZERO_START_DOCKER.txt

Ключевые моменты:
- Перед init SQL обязательно дождаться готовности ClickHouse:
  do { Start-Sleep -Seconds 2; docker compose --env-file .env exec -T clickhouse clickhouse-client --query "SELECT 1" } while ($LASTEXITCODE -ne 0)
- Сначала 01_init.sql, потом 02_mart.sql
- Затем generator -> loader -> producer
- Потом пауза 5 сек и повторный запуск 02_mart.sql для актуального snapshot качества

8) Частые проблемы и как чинить
====================================================================

Проблема: Connection refused (localhost:9000) при init SQL
Причина: ClickHouse еще не готов после up -d.
Решение: добавить цикл ожидания SELECT 1 перед init.

Проблема: Unknown table probablyfresh_raw.* при 02_mart.sql
Причина: не применился 01_init.sql.
Решение: применить 01_init.sql, затем 02_mart.sql.

Проблема: ETL выгружает только header (0 строк)
Причина: пустой MART.
Решение: восстановить init SQL и прогнать producer, проверить counts.

Проблема: "No data" в Grafana
Причина: нет данных в таблицах или datasource/plugin не поднялись.
Решение:
- проверить counts в ClickHouse,
- проверить datasource clickhouse_raw,
- при необходимости restart grafana.

Проблема: долгое ожидание на pyspark/jdbc каждый запуск
Причина: runtime install/download.
Решение: уже исправлено кастомным app image + fat JDBC jar.

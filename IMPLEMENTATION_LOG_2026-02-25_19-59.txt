1) Что реализовано на текущий момент
====================================================================

Реализован полный контур платформы:

1. Data pipeline (core):
   - Генерация JSON данных (stores, products, customers, purchases).
   - Загрузка в MongoDB.
   - Поток MongoDB -> Kafka -> ClickHouse RAW.
   - Трансформация RAW -> MART через Materialized Views.
   - ETL (PySpark) -> CSV -> S3.

2. RAW слой ClickHouse:
   - probablyfresh_raw.stores_raw
   - probablyfresh_raw.products_raw
   - probablyfresh_raw.customers_raw
   - probablyfresh_raw.purchases_raw
   - probablyfresh_raw.purchase_items_raw
   - Kafka engine + MV ingestion настроены.

3. MART слой ClickHouse:
   - probablyfresh_mart.stores_mart
   - probablyfresh_mart.products_mart
   - probablyfresh_mart.customers_mart
   - probablyfresh_mart.purchases_mart
   - probablyfresh_mart.purchase_items_mart
   - probablyfresh_mart.mart_quality_stats

4. Data Quality и SMART-валидации:
   - trim/lower для ключей.
   - проверки форматов ID через regex:
     store_id: ^store-[0-9]{3}$
     product_id: ^prd-[0-9]{4}$
     customer_id: ^cus-[0-9]{6}$
     purchase_id: ^ord-[0-9]{6}$
   - проверка дат (birth_date <= today, purchase_dt <= now).
   - проверки количественных полей items (quantity > 0, total_price >= 0).
   - quality snapshots в mart_quality_stats.

5. PII соответствует ТЗ (one-way anonymization):
   - Нормализация email/phone сохраняется.
   - Хранение в payload полей email_enc / phone_enc.
   - Обезличивание через SHA-256:
     sha256(salt + ":" + normalized_value).hexdigest()
   - salt берется из PII_HASH_SALT (>=16 символов).
   - В MART добавлена валидация хэшей (hex64) для email/phone.

6. ETL витрина признаков:
   - jobs/features_etl.py использует customers_mart, purchases_mart, products_mart, purchase_items_mart.
   - Реализованы 30 бинарных фич (включая no_purchases, bought_milk_last_30d, vegetarian_profile и др.).
   - Выгрузка результата в один CSV в S3: analytic_result_YYYY_MM_DD.csv.

7. Оркестрация:
   - Airflow поднят в compose (LocalExecutor).
   - DAG etl_to_s3_daily:
     wait_clickhouse -> run_etl (в app через docker exec) -> verify_minio/S3 object.
   - Web UI Airflow на localhost:8080.

8. Monitoring / alerting:
   - Grafana + ClickHouse datasource provisioning.
   - Dashboard ProbablyFresh RAW Overview.
   - Alerting + Telegram contact point/provisioning.

9. API слой (Django + DRF):
   - REST API для вкладок Overview / Pipelines / Data Quality / Exports / Settings.
   - Actions endpoints с запуском jobs и polling run status.
   - Token auth, CORS, OpenAPI docs.

10. Frontend (React + TS + Tailwind):
   - 5 страниц control panel.
   - Интеграция с backend API.
   - RU/EN переключение, dark/light, toasts с auto-dismiss.
   - Safe Mode сохраняется и блокирует Refresh MART.

4) Текущий источник истины по данным
====================================================================

1. Overview KPI (UI):
   - читает backend /api/overview/kpis
   - метрики строятся по MART FINAL (uniq и final counts).

2. Data Quality (UI):
   - читает backend /api/quality/*
   - метрики строятся из mart_quality_stats (snapshot по RAW/MART).

3. Grafana RAW dashboard:
   - использует запросы к RAW и mart_quality_stats.

Важно:
- Показатели в Overview, Data Quality и Grafana могут отличаться по смыслу:
  Overview = uniq/final
  Quality = total/duplicates snapshot
  Grafana = mix RAW counters + quality rows

6) Runbook: запуск data-контура (Docker-only)
====================================================================

Основной runbook: ZERO_START_DOCKER.txt

Ключевые принципы:
- .env НЕ перезаписывать, если уже существует.
- before init SQL: дождаться ClickHouse SELECT 1.
- порядок init: 01_init.sql -> 02_mart.sql.
- после producer дать паузу и повторить 02_mart.sql для актуального quality snapshot.
- ETL запускать через app контейнер (spark-submit).
- Airflow команды:
  - airflow dags unpause etl_to_s3_daily
  - airflow dags trigger etl_to_s3_daily
  - airflow dags list-runs -d etl_to_s3_daily

10) Известные моменты / зоны внимания
====================================================================

1. Grafana панель качества может брать "последнюю строку" mart_quality_stats без фильтра entity,
   из-за чего отдельные single-stat панели могут показывать метрику другой сущности.

2. Разница цифр между вкладками может быть нормальной:
   - Overview показывает deduped/uniq MART,
   - Data Quality показывает snapshot total/duplicates,
   - Grafana часть панелей показывает RAW totals.

3. Если нужны одинаковые цифры везде, нужно унифицировать definition-of-metric
   (например везде purchases FINAL uniq, либо везде latest quality purchases).

11) Рекомендуемый порядок проверки за 5 минут
====================================================================

1. Поднять core сервисы + дождаться ClickHouse.
2. Применить 01_init.sql и 02_mart.sql.
3. generate_data -> load_to_mongo -> produce_from_mongo --once.
4. Подождать 5 сек и повторно применить 02_mart.sql.
5. Проверить counts RAW/MART и mart_quality_stats.
6. Запустить ETL и проверить объект analytic_result_YYYY_MM_DD.csv в S3.
7. Открыть Grafana (3000), Airflow (8080), Frontend (5173).
8. В UI проверить One-click actions и polling run status.

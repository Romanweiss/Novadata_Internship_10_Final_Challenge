# ProbablyFresh: варианты расширения проекта и практический план

## 1. Куда расширять проект дальше

Ниже приоритетный roadmap от самого полезного к менее критичному.

### 1.1 Data Contract + ingest как сервис (высокий приоритет)
- Ввести явные версии входного контракта (`schema_version`) для загружаемых данных.
- Добавить слой `landing/staging` перед Mongo/Kafka, чтобы новые форматы не ломали текущий RAW.
- Реализовать маппинг полей по правилам (preset), а не жёстко в коде.
- Добавить отчёт качества импорта: сколько строк валидно/невалидно, какие причины отклонений.

### 1.2 Надёжность пайплайна
- Идемпотентность запусков (`run_id`, dedup-key, безопасный retry).
- Повторные попытки для сетевых операций (S3/Airflow/Telegram) с backoff.
- SLA/timeout-политики на каждый job.

### 1.3 Data Quality 2.0
- Расширить проверки: диапазоны, допустимые enum, длины ID, cross-table проверки (FK-подобные).
- Отдельная таблица ошибок качества (`dq_errors`) с примерами строк.
- Авто-алерты не только по duplicates, но и по invalid_ratio, late data, missing partitions.

### 1.4 Наблюдаемость
- Метрики pipeline в Prometheus/Grafana (duration, throughput, failed_runs).
- Корреляция job run -> лог -> экспорт (через `run_id`).
- Datasets freshness dashboard (сколько минут назад обновлялась каждая таблица).

### 1.5 Безопасность и governance
- RBAC для действий в UI (кто может запускать destructive job).
- Audit trail по пользователям: кто запустил, когда, с какими параметрами.
- Централизованные секреты (не хранить чувствительные ключи в открытых env-файлах).

### 1.6 CI/CD и тестирование
- Contract tests на входные схемы.
- Smoke + integration тесты на весь контур Docker.
- Автопроверка SQL (ClickHouse init scripts) и миграций Django.

## 2. Что реально доступно через Django Admin уже сейчас

Если в проекте уже зарегистрированы текущие сервисные модели, через админку можно:

### 2.1 Операции и мониторинг
- Просматривать `JobRun` (очередь/успех/ошибка, duration, хвост логов, error message).
- Просматривать `ExportAudit` (какой объект выгружен в S3, размер, статус, время).
- Просматривать `AlertEvent` (когда и почему ушёл alert, success/fail, payload).
- Управлять `PipelinePreset` (готовые конфиги ручных запусков для UI/API).

### 2.2 Администрирование
- Управлять пользователями/правами Django (`auth` + группы).
- Выдавать/отзывать DRF token’ы (если используется TokenAuth).
- Фильтровать, искать, экспортировать записи админки.

### 2.3 Что стоит добавить в Admin для удобства
- Кастомные list filters: `status`, `job_name`, период времени.
- Admin actions:
  - «Re-run from preset»
  - «Mark as acknowledged» для alert событий
  - «Open linked object in S3» (через presigned URL/метаданные)
- Read-only поля для системных атрибутов (timestamps, duration, log_path).

## 4. Можно ли добавить кнопку в UI для загрузки данных

Да, это практично и рекомендуется.

### 4.1 Целевой поток (без ломки текущего пайплайна)
1. В UI: кнопка `Upload Data` (например, на вкладке Pipelines или Exports).
2. Frontend отправляет файл в backend endpoint (`POST /api/imports`).
3. Backend создаёт `ImportBatch` в статусе `queued`.
4. Асинхронный worker (или фоновой job runner) парсит файл.
5. Валидные строки идут в `landing`/`staging` и дальше в существующий pipeline.
6. Невалидные строки пишутся в `ImportRowError`.
7. UI показывает прогресс и отчёт импорта.

### 4.2 Минимальные API для этого
- `POST /api/imports` — загрузка файла + выбор сущности + preset.
- `GET /api/imports/{id}` — статус и статистика.
- `GET /api/imports/{id}/errors` — ошибки по строкам.
- `POST /api/imports/{id}/apply` — подтвердить продвижение в RAW (если нужен human approval).

## 5. Что делать, если структура загружаемых данных не совпадает

Это нормальный и частый кейс. Нужен «контракт + маппер + карантин».

### 5.1 Правила совместимости
- **Обязательные поля**: без них запись не проходит.
- **Опциональные поля**: можно заполнять default/null.
- **Переименованные поля**: через маппинг preset (`customerId -> customer_id`).
- **Новые поля**: сохранять в `extra_payload` без падения пайплайна.
- **Неизвестный формат**: batch помечать как `failed`/`partial`, не продвигать в MART.

### 5.2 Техническая стратегия
- Ввести `schema_version` у каждой загрузки.
- Хранить маппинги в `PipelinePreset` или отдельной `SchemaMapping` модели.
- Прогонять валидацию по правилам (pydantic/jsonschema/custom validators).
- Разделять статусы:
  - `accepted` — можно грузить дальше,
  - `accepted_with_warnings` — допустимо с потерями,
  - `rejected` — требуется исправление.

### 5.3 Почему это важно
- Не ломается текущий RAW/MART при изменении формата со стороны источника.
- Есть прозрачный отчёт, почему конкретные строки не приняты.
- Можно быстро адаптироваться к новому поставщику/формату без переписывания всего кода.

## 6. Практический минимум, который стоит сделать первым

1. Добавить `ImportBatch`/`ImportRowError` и endpoint загрузки файла.
2. Реализовать 1 preset маппинга для `purchases`.
3. Добавить базовую схему валидации + карантин ошибок.
4. Вывести в UI отчёт импорта (valid/invalid + top причин).
5. Подключить алерт на высокий процент rejected строк.

## 7. Вывод

Проект уже близок к production-подходу по ядру pipeline. Следующий реальный шаг — сделать управляемый ingestion через UI + устойчивую обработку schema mismatch. Это даст максимальную пользу: меньше ручных операций, меньше риска поломок, выше прозрачность для команды и заказчика.
